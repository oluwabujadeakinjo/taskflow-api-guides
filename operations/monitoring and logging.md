# Monitoring and logging for TaskFlow API
This document should teach developers/system administrators how to:
- monitor the API;
- inspect logs;
- diagnose failures; and
- verify service health.

## Prerequisites
You need to have:
- Docker installed
- PM2 (Process Manager 2) installed
- Deployed TaskFlow API
- Terminal access
- cURL installed
- Administrator permissions

## Health monitoring
A health check determines if an API is operational, functional, and able to fulfil requests. It is done by sending a request to the `/health` endpoint.

**Health monitoring can be done by:**
- **Uptime verification:** This continously proves the API is reachable and operational from external users. It triggers an alert if the API fails.

- **Service availability:** Measures the percentage of time the API is fully functional and able to successfully process requests, over a specific period of time.

### Request example

```Bash
# Ping the health check endpoint
curl http://localhost:3000/health
```

### Expected response
A healthy API typically responds with a `200 OK` status code and a json message:

```JSON
{
    "status": "healthy",
    "timestamp": "2026-05-13T22:50:07Z",
    "uptime_seconds": 172800
}
```

## PM2 monitoring
### Step 1: Check the process status
The process status tells the current operating state of a managed application:

```Bash
# Check the status of the project
pm2 status
```
This table shows the dashboard of all managed applications:

|**Property**|**Description**|
|------------|---------------|
|**id**|Unique numeric identifier assigned to the process by PM2.|
|**name**|String name given to the application.|
|**namespace**|Grouping category used to organize multiple processes.|
|**version**|The current version of the deployed application|
|**mode**|The execution mode.|
|**pid**|The current OS process ID assigned to the application.|
|**uptime**|Continuous duration the current process instanc has been running.|
|**↺**|Total number of times the process has restarted due to crashes or manual commands.|
|**status**|Current operational state e.g., `online`, `stopped`, etc.,|
|**cpu**|Percentage of CPU core capacity currently consumed by process.|
|**mem**|Total number of RAM currently consumed by the process.|
|**user**|The operating system user account running the PM2 process instance.|

### Step 2: Enable real-time dashboard
Use the `monit` command for a continuous, real-time dashboard of resources:

```Bash
# Enable real-time dashboard of resources
pm2 monit
```

## Viewing application logs
Application logs are timestamped records of events, actions, and errors generated by applications during their operation. 

### Understanding log streams
PM2 automatically captures and separates two primary output streams of the application:

|**Stream**|**Log File**|**Description**|
|----------|------------|---------------|
|**stdout**|`out.log`|**Standard Output:** General application events for routine tracking.|
|**stderr**|`err.log`|**Standard Error:** Critical issues, stack traces, and unhandled exceptions.|

### View combined real-time logs
To see both streams merged into a single real-time feed, use:

```Bash
pm2 logs taskflow-api
```

## Log streaming with Docker
To view logs generated by applications running inside Docker containers, use the standard Docker command-line interface.

### Step 1: Identify the running container
To find the specific identifier for the target container, use:

```Bash
docker ps
```

### Step 2: View the logs
Choose one of these two methods to view the logs:

**Method A: View historical logs**
To view the backlog of messages that the application has written to `stdout` or `stderror`, fetch the static snapshot:

```Bash
docker logs <container-id>
```

**Method B: Stream logs**
To follow logs in real-time, stream the output:

```Bash
# -f or --follow flag aids in log streaming
docker logs -f <container-id>
```

## Identifying operational red flags
If you notice any of the following indicators in your `pm2 status` or `pm2 monit` dashboard, the application requires immediate attention:

|**Indicator**|**Observation (PM2/Logs)**|**Root Cause & Action**|
|-------------|--------------------------|-------------|
|**Repeated Crashes**|The `status` toggles between `online` and `errored`.|Likely a syntax error, missing `.env` variable, or an unhandled exception. Change `pm2 logs --err`.|
|**Increasing Memory**|The `mem` value rises steadily without dropping.|The application is holding onto data it should release. Investigate global variables or unclosed database cursors.|
|**Slow Response Times**|The `/health` ping takes >500ms to respond.|The CPU is pegged or the event loop is blocked. Check `cpu` usage in `pm2 monit`.|
|**Failed DB Connections**|Log stream shows `ECONNREFUSED` or `Timeout`.|The PostgreSQL service is down, or the connection pool is exhauseted. Restart the DB service.|
|**Excessive Restarts**|High numeric value in the `↺` column.|Even if the app is "online" now, it has crashed many times. Compare `uptime` with the restart count to see frequency.|

## Operational troubleshooting
Use the following matrix to diagnose and resolve issues:

|**Issue**|**Diagnostic Observation**|**Realistic Solution**|
|---------|--------------------------|----------------------|
|**API returns 500 errors**|`pm2 logs` shows `TypeError` or `UnhandledPromiseRejection`.|Check code and logic. Wrap the failing logic in a `try...catch` block and redeploy.|
|**Container repeatedly exits**|`docker ps -a` shows status `Exited (1)`.|Ensure your `CMD` in the Dockefile points to a valid script. Verify the `,env` is currently mounted inside the container.|
|**PM2 process offline**|`pm2 status` shows `stopped` or `errored`.|Run `pm2 restart taskflow-api`. If it fails again immediately, run `pm2 show taskflow-api` to see the exact error path.|
|**Missing logs**|`pm2 logs` is empty or not updating.|The disk might be full, or the log file permissions have changed. Run `pm2 flush` to clear state handles or check `/home/user/.pm2/logs`.|
|**Disk space issues**|`no space left on device` error in terminal.|Run `docker system prune` to remove old images/containers and use `pm2 install pm2-logtotate` to prevent logs from growing indefinitely.|

## Security recommendations
- Restrict log access to admin.
- Avoid including densitve data in logs.
- Rotate logs.
- Monitor failed login attempts.
